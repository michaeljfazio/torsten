### Lookup (500 lookups per iteration)

| Size | In-Memory | Mmap | Mmap/InMem Ratio |
|------|-----------|------|-----------------|
| 1K | 8.97 µs | 2.36 µs | 0.26x |
| 5K | 9.03 µs | 3.09 µs | 0.34x |
| 10K | 9.12 µs | 2.83 µs | 0.31x |
| 25K | 9.40 µs | 2.33 µs | 0.25x |
| 50K | 9.44 µs | 2.17 µs | 0.23x |
| 100K | 9.68 µs | 2.16 µs | 0.22x |

**Observation**: Mmap lookup is **3-4x faster** than in-memory HashMap! This is because the mmap hash table uses direct array indexing with linear probing, whereas HashMap has more overhead (hashing + bucket resolution + pointer chasing). Mmap lookup is essentially flat regardless of dataset size. In-memory shows slight degradation at 100K (9.68µs vs 8.97µs at 1K).

## Key Takeaways

1. **Mmap block index is the clear winner for lookup-heavy workloads** — 3-4x faster lookups, 4x faster open at scale. The insert overhead (3x slower) is acceptable since inserts happen infrequently (block arrival) while lookups happen constantly.

2. **Default LSM config is well-tuned** — no significant improvement from varying memtable/cache/bloom at 10K scale. Differences will matter at millions of entries.

3. **UTxO store scales well** — near-linear insert and scan performance, sub-linear lookup degradation (only 40% slower at 100x dataset size).

4. **Mmap should be the default for `low-memory` profile** — the open-time improvement alone justifies it, and lookups are faster too.

5. **For `high-memory` profile, mmap is also beneficial** for lookup speed. Consider making mmap the default for both profiles, keeping in-memory as a fallback option.
